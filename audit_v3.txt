[CRIT-3] Google Play publish ALWAYS FAILS with 403 error  ← ALL PUBLISHES BROKEN
  Error  : "This app does not have a title set for the default language."
  Logs   : celery-worker shows this error for suggestions 19, 20, 21, 25, 26
  DB     : 5 suggestions with status=approved, publish_status=failed
  Problem: When publishing to Google Play API, the edit commit is failing with:
           HttpError 403: "This app does not have a title set for the default language."
           This means the Google Play listing for com.NetSafe.VPN does NOT have
           a "title" field set in the default language in the edit being committed.
           The code sends only the fields it wants to update (e.g. long_description)
           but Google Play requires ALL required fields (title, short description)
           to be present when committing an edit.
  Root cause in code:
    app/services/execution.py — when creating a Google Play edit, the code
    only sets the specific field being changed (long_description, short_description).
    It does NOT fetch the existing title first and include it in the edit.
    Google Play API: when you commit an edit, ALL fields that are not explicitly
    set in the edit will be CLEARED. If title is not set in the edit, Google
    blocks the commit with 403.
  Fix    :
    In execution.py publish logic, before committing a listing edit:
    1. First GET the existing listing: GET .../listings/{language}
    2. Merge the new field change ON TOP of the existing listing data
    3. Then PUT the merged listing (with title included)
    4. Then commit the edit
    Example fix in execution.py:
      existing = service.edits().listings().get(
          packageName=package, editId=edit_id, language=language).execute()
      merged = {**existing, field_name: new_value}
      service.edits().listings().update(
          packageName=package, editId=edit_id, language=language,
          body=merged).execute()

[BUG-4] Duplicate pipeline tasks sent by Celery beat
  Logs   : daily_pipeline[dc729353-f4c1-4f58-9e6d-2027971f1bca] received at:
           11:56, 12:56, 13:56, 14:56, 15:16
           Same task_id queued multiple times across hours.

           instances are still "running" (zombie). Beat doesn't check if
           a pipeline for that app is already running before dispatching.
  Fix    :
    In dispatch_daily_pipeline task:
      from redis import Redis
      r = Redis.from_url(settings.redis_url)
      lock_key = f"pipeline_running:app_{app_id}"
      if r.get(lock_key):
          logger.info(f"Pipeline already running for app {app_id}, skipping")
          return
      r.setex(lock_key, 3600, "1")  # 1 hour TTL
      try:
          run_pipeline(app_id)
      finally:
          r.delete(lock_key)

[BUG-8] sqlalchemy InFailedSqlTransaction error in worker
  Logs   : celery-worker shows:
           "sqlalchemy.exc.InternalError: current transaction is aborted,
            commands ignored until end of transaction block"
           This happens when a DB query fails mid-transaction and the
           session is not properly rolled back.
  Problem: In app/workers/tasks/, the sync DB sessions don't have proper
           exception handling. If one query fails, subsequent queries in
           the same session fail with "InFailedSqlTransaction".
  Fix    :
    In all worker task files using sync sessions:
      try:
          db.execute(...)
          db.commit()
      except Exception:
          db.rollback()
          raise

      → publish_status = "published"

    MODE 3 — "Auto to Publish" (NEW)
      → No manual approval step needed
      → Auto-approve rules from auto_approve_rules table run
      → If suggestion passes rules → automatically queued for publish
      → Combined with MODE 1 or MODE 2
      → Controlled by: manual_approval_required = false in global_config
        (this key already exists in DB! Just needs UI toggle)

  Dashboard UI:
    Add a 3-segment button below pipeline status section:
    [ Soft Publish ] [ Live to Google ] [ Auto to Publish ]
    Active mode highlighted. Clicking changes mode with confirmation dialog.
    Also shown as badge in Dashboard header (replacing or alongside DRY RUN badge).

  Backend changes needed:
    1. Add "publish_mode" key to global_config:
       "soft" = soft publish (draft only), "live" = full publish
    2. Add endpoint: PATCH /api/v1/settings/publish-mode
       body: {"mode": "soft" | "live", "auto_approve": true | false}
    3. In execution.py:
       if publish_mode == "soft":
           create edit + update fields → DO NOT call edits().commit()
           set publish_status = "soft_published"
       if publish_mode == "live":
           existing behavior (create + update + commit)
    4. publish_status should be expanded to include "soft_published"
